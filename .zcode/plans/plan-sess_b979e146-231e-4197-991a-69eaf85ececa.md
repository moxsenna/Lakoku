Tujuan: potong delay 2–3 detik saat pindah menu web dengan fokus pada root cause terbesar lebih dulu, lalu tambahkan perceived-performance agar klik terasa instan.

Rencana implementasi

1. Sempitkan middleware ke route yang benar-benar butuh auth
- Ubah `middleware.ts` matcher agar tidak menangkap hampir semua route publik.
- Target awal: lindungi hanya route yang memang privat seperti `/baca/:path*`, `/akhir/:path*`, `/koleksiku/:path*`, dan route auth callback bila perlu.
- Pertahankan logika redirect auth di `lib/supabase/proxy.ts`, tapi kurangi frekuensi eksekusi karena sekarang middleware hanya jalan di route privat.
- Dampak: klik menu publik tidak lagi selalu kena `supabase.auth.getUser()` di middleware.

2. Kurangi `force-dynamic` pada route publik
- Audit dan hapus `export const dynamic = 'force-dynamic'` dari page yang tidak wajib realtime per request.
- Prioritas hapus dari:
  - `app/beranda/page.tsx`
  - `app/cerita/[id]/page.tsx`
  - `app/page.tsx`
  - `app/mulai/page.tsx` bila aman
- Pertahankan dynamic hanya pada route yang memang tergantung session/cookies atau data yang harus selalu fresh, seperti page reader/pembayaran bila perlu.
- Dampak: Next bisa prefetch dan cache route payload lebih efektif.

3. Tambah cache untuk data publik yang sering dibaca
- Bungkus query publik di `lib/api/queries.ts` dengan primitive cache yang sesuai pola Next 16/React server cache.
- Kandidat awal:
  - `queryStories()`
  - `queryStory()`
  - `queryChapter()`
  - `queryLatestAvailableChapter()`
  - `getReadingPolicy()` di `lib/credits/server.ts`
- Pisahkan data publik cacheable dari overlay state user agar route publik tetap cepat.
- Dampak: klik antar menu tidak selalu hit Supabase untuk data cerita yang sama.

4. Hilangkan bottleneck N+1 di halaman profil
- Refactor `app/profil/page.tsx` agar tidak melakukan `Promise.all(stories.map((s) => getStory(s.id)))`.
- Hitung `totalPilihan` dari data yang sudah tersedia jika cukup, atau buat helper agregat tunggal yang mengambil hanya field yang dibutuhkan sekali jalan.
- Kurangi panggilan berulang ke `getReaderState()` dan `auth.getUser()` dari jalur render profil.
- Dampak: buka Profil jadi jauh lebih ringan dan skalanya tidak memburuk saat jumlah cerita bertambah.

5. Kurangi fetch duplikat di reader path bila mudah dilakukan aman
- Tinjau `lib/api/server.ts#getChapter()` yang memanggil `getStory()` lagi saat `chapterNumber` tidak ada.
- Bila page pemanggil sudah punya `story`, teruskan chapter target atau sediakan helper yang menerima current chapter agar tidak query story dua kali.
- Ini bukan prioritas pertama untuk menu switching, tapi bagus untuk mengurangi delay saat masuk reader.

6. Tambah `loading.tsx`/pending UI untuk route utama
- Tambahkan route-level loading untuk page yang masih server-heavy, minimal:
  - `app/beranda/loading.tsx`
  - `app/koleksiku/loading.tsx`
  - `app/profil/loading.tsx`
- Reuse pola skeleton/pending yang sudah ada dari `components/mulai/onboarding-flow.tsx` dan spinner dari `components/ui/spinner.tsx` agar gaya konsisten.
- Dampak: walau backend masih butuh waktu, klik terasa responsif dan tidak terlihat seperti tombol macet.

7. Verifikasi hasil dengan pengukuran sederhana
- Bandingkan sebelum/sesudah pada flow menu bawah: `/beranda`, `/koleksiku`, `/mulai`, `/profil`.
- Cek apakah request middleware berkurang, halaman publik tidak lagi selalu dynamic, dan Profil tidak lagi melakukan fanout query berat.
- Jika tersedia, jalankan smoke/lint yang relevan untuk memastikan refactor aman.

Urutan eksekusi yang disarankan
1. Middleware scope
2. Hapus `force-dynamic` route publik
3. Refactor Profil N+1
4. Tambah cache publik
5. Tambah `loading.tsx`
6. Rapikan fetch duplikat reader

Hasil yang diharapkan
- Klik menu publik jauh lebih cepat karena tidak selalu auth roundtrip + full dynamic render.
- Profil tidak lagi jadi halaman paling berat.
- UX terasa lebih instan walau beberapa route masih server-rendered.
- Perubahan tetap minimal dan mengikuti pola codebase saat ini, bukan rewrite besar.
